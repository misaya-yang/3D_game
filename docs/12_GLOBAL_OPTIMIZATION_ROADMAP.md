> 版本：v1.0 · 2026-07-16  
> 目的：把“全面优化到真正可玩”变成有限、可追踪、可验收的串行工作，禁止凭感觉跨系统乱改。  
> 当前权威指针仍为 [`10_PROGRESS.md`](10_PROGRESS.md)；本文件负责全局轮次，不替代系统规格与 Goal 卡片。

## 1. 总完成定义

长期优化任务只有同时满足下列条件才能标记完成：

1. `G09-01`—`G09-10` 全部为 `done`，且每一张卡都有真实 Player 和自动化证据。
2. 干净存档关键旅程“启动 → 灵根 → 青石任务/战斗/成长 → 苍梧 → 黑风洞 → 死亡恢复 → 存读档”连续 3 次完成。
3. 全局问题台账没有未归属项，P0（崩溃/损坏）和 P1（软锁/无法推进/核心输入失效/主要视觉阻断）均为 0。
4. UI 三视口与键鼠/手柄主流程可达；角色、敌人和三图主题在正常游戏距离可辨；核心操作都有明确反馈。
5. 全量 EditMode/PlayMode、macOS Release、Boot/三图/战斗/存读档冒烟和既有性能预算全部通过。

测试通过只证明其覆盖范围，不自动证明“游戏已好玩”。每轮仍必须保留同路径 Player 基线、改后证据和明确问题台账。

## 3. 每轮固定流程

每一轮严格执行六步；缺一步就不能改为 `done`：

1. **基线**：在固定视口、存档、地点和输入状态下采集 Player 截图、日志、状态与性能。
2. **冻结问题集**：写清问题 ID、严重度、复现步骤、期望、证据、归属 Goal；P2/P3 不偷塞进当前轮。
3. **最小成组实现**：按问题因果关系修复，不做与当前 acceptance 无关的“顺手重构”。
4. **同路径对照**：用与基线相同的旅程、视口和状态重拍；视觉截图必须成对比较。
5. **门禁**：目标测试 → 必要回归 → 全量测试 → Release 构建 → 对应 Player 冒烟。
6. **关账与切换**：更新问题台账、证据、Goal 和 `10_PROGRESS.md`；当前轮 `done` 后才领取下一轮。

## 4. 问题严重度与路由

| 级别 | 定义 | 处理规则 |
|------|------|----------|
| P0 | 崩溃、存档损坏、无法启动、持续数据丢失 | 当前轮立即修复并新增回归，禁止带入下一轮 |
| P1 | 软锁、主线无法推进、核心输入失效、主要画面/相机阻断 | 当前轮修复并提供前后证据 |
| P2 | 明显降低理解、手感、视觉或效率，但有可靠绕行 | 路由到最早匹配的后续 Goal，不跨轮扩容 |
| P3 | 轻微润色、偏好或非核心边角 | 只登记；总签核前决定修复或明确接受 |

问题台账至少包含：`id`、`goal`、`severity`、`state`、`repro`、`expected`、`evidence_before`、`fix`、`evidence_after`、`regression`。没有基线和验收定义的问题，不允许直接改代码。

## 5. 防盲改规则

- 同一时刻只有一个 `in_progress` Goal；路线图不能成为并行开工许可。
- 视觉更改必须从真实 Player 同路径截图出发，并在相同视口复核；不以 Scene 视图或代码意图代替。
- 玩法更改必须先证明当前行为与权威规格冲突；数值只改对应权威章节。
- 第三方资产只接受许可明确、实际使用、可重复下载并有 SHA-256 的精选子集。
- 任何新增预估超过 1 小时的工作必须成为独立 Goal 或调整当前卡片，不在问题修复中暗增范围。
- 目标测试、全量测试、构建或冒烟任一失败，当前轮保持 `in_progress`；先诊断，不回滚历史 `done`。
